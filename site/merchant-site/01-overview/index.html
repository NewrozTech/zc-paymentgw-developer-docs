    1. Overview¶

    What the gateway does¶

    The ZiCharge Merchant Payment Gateway lets your platform accept payments from any ZiCharge wallet account. From the customer's perspective, it is a one-page checkout where they enter their mobile number and password (or 4-digit PIN) and confirm. From your platform's perspective, it is a small set of HTTP endpoints plus an asynchronous callback (IPN) delivered to a URL you configure.

    The gateway is engineered on a Java 21 / Spring Boot 3.2 backend. Highlights that matter to integrators:

    • Stable, versioned contract — the response envelope (messages, data, code) and field names are guaranteed within a major version.
    • Idempotent token lifecycle — minting a token, paying it, cancelling it, and retrying an IPN are all safe to re-issue without producing duplicate state.
    • Race-free state transitions — the payment record uses pessimistic locking and conditional updates so concurrent pay / cancel / expiry resolve to a single, consistent outcome.
    • At-least-once IPN delivery with automatic retries — IPN delivery is cooldown-gated and replayable; you will receive every terminal event eventually, even if your listener is briefly unreachable.
    • SSRF-hardened callbacks — outbound IPN traffic is restricted by an allow-list policy on the gateway side.
    • Stateless, horizontally scalable JVM tier — under a TLS-terminating load balancer with a Web Application Firewall.

    Supported flows¶

    Flow What it is Best for
    Hosted Payment Page Mint a token → redirect customer to the gateway's hosted page → IPN + validation Web checkouts, mobile webviews, QR-code-to-link
    Direct Charge Mint a token → submit customer credentials to POST /merchant/payment/direct from a trusted channel Custom checkout surfaces under strict PCI-equivalent controls
    QR Token Same as hosted page, but the response also includes a server-generated QR URL In-store POS, kiosks, printed media
    Cash-Back Push a refund / promo credit from the merchant wallet to a customer wallet Refunds, loyalty rewards

    All four flows share the same authentication and the same IPN contract.

    Environments¶

    Environment Gateway base URL Purpose
    Sandbox https://dev.zicharge.com Integration testing, no real funds movement
    Production https://secure.zicharge.com Live transactions

    All traffic is HTTPS-only with TLS 1.2 or higher. HTTP downgrade is not supported.

    API path conventions¶

    Two equivalent path prefixes are exposed:

    • /api/v3/merchant/... — the canonical, versioned path. Recommended for new integrations.
    • /merchant/... — a flat path retained for backwards compatibility with integrations that started on the previous gateway. No behavioural difference.

    Both prefixes serve the same controllers, the same validation, and the same response envelope. Pick one and stick with it per integration.

    Content types¶

    Endpoint family JSON application/x-www-form-urlencoded multipart/form-data
    Generate token ✅ ✅ ✅
    Validate / status / token-data ✅ ✅ ✅
    Direct payment ✅ ✅ ✅
    Cash-back ✅ ✅ ✅
    Transactions list ✅ (GET) n/a n/a
    IPN delivered to your listener n/a ✅ (we POST form-urlencoded to you) n/a

    JSON is preferred for new integrations. Form-encoded variants exist so that integrations that were originally form-based keep working without changes.

    Response envelope¶

    Every endpoint returns HTTP 200, including when the outcome is a business-level failure. The real outcome is carried inside the JSON body in the code field:

    {
  "code": 200,
  "messages": ["Successfully Deposited."],
  "data": { ... }
}
    Field Type Meaning
    code integer The logical outcome. 200 = success. Non-success values are documented in 06-error-codes.md.
    messages array of strings One or more localized human-readable messages. Always an array.
    data object | array | null Endpoint-specific payload. null on errors.

    Why HTTP 200 instead of HTTP 4xx? This contract is identical to the previous gateway and is preserved on purpose. Some legacy proxies and merchant SDKs treat non-2xx HTTP statuses as transport failures and retry, which can cause duplicate side-effects. Keeping the logical outcome in the body lets your client decide retry semantics safely.

    Endpoints that pre-date the v3 prefix (POST /merchant/generate-payment-token, POST /merchant/payment/direct) use an extended legacy envelope that adds top-level fields (token, transactionInfo) alongside messages and code. The exact shape is documented per-endpoint in 04-api-reference.md.

    Amounts and currency¶

    • All amounts are in IQD (Iraqi Dinar).
    • Amounts are transmitted as decimal numbers; integer amounts are required for transaction limits and are recommended for bill_amount.
    • The minimum bill_amount is 250 IQD at token generation. Sub-minimum tokens are rejected before they are created.
    • Amounts are validated with BigDecimal precision on the server. There is no floating-point rounding in the payment path.

    Locale & internationalization¶

    • The gateway honours the Accept-Language request header.
    • Supported languages: English (en), Arabic (ar), Kurdish (ku).
    • Some endpoints also accept a lang field in the body for explicit override; when both are provided, the body field wins.
    • Error messages, success messages, and the hosted payment page are all localized.

    Identifiers used in this API¶

    Identifier Format Source Use
    merchant_mobile_no +964 + 10 digits, optional leading 0 inside the local part Provisioned at merchant onboarding Identifies the merchant wallet on every server-to-server call
    store_password Opaque string Provisioned at merchant onboarding Authenticates server-to-server calls; never exposed to customers
    order_id Opaque string, must be unique per merchant Your system Idempotency anchor — same order_id on token mint won't double-charge
    payment_token Opaque token, ≥ 32 chars Returned by generate-payment-token Bound to one order_id and one validity window
    transaction_id Numeric, server-issued Returned in IPN and validation responses Use this in your books as the canonical ZiCharge reference
    customer_mobile_no +964 + 10 digits Set by the customer at checkout, optionally pre-filled The paying account

    Token lifecycle (at a glance)¶

       created           paid           validated
  ─────────►───────────────►────────────────►
                 ▲                ▲
       cancelled │                │ fetch-status / validate-payment
       (user / system / timeout)  │
                 │                │
                 └─► IPN dispatched (Success | Cancelled)

    A token is valid for 15 minutes from creation. After 15 minutes it can no longer be paid. If a customer opens an already-expired token, the gateway will mark it cancelled and fire a Cancelled IPN so your system can release the cart.
