    2. Authentication & Security¶

    The gateway exposes three distinct trust boundaries. Make sure you understand which one each call uses before you ship.

    ┌────────────────────────────────────────────────────────────────────┐
│ 1. Server-to-Server (merchant backend → ZiCharge)                  │
│    merchant_mobile_no + store_password                             │
│    HTTPS + (optional) IP allow-list                                │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│ 2. Customer-facing (customer browser → ZiCharge hosted page)       │
│    Bearer token (`payment_token`) in URL query                     │
│    Customer authenticates with their own wallet mobile + password  │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│ 3. Asynchronous (ZiCharge → merchant IPN listener)                 │
│    Form-POST to merchant-configured ipn_url                        │
│    Recommended: HMAC verification on receipt (see §2.4)            │
└────────────────────────────────────────────────────────────────────┘

    2.1 Merchant credentials¶

    Two values, both provisioned at merchant onboarding through the ZiCharge merchant dashboard:

    Credential Purpose
    merchant_mobile_no Public identifier of your merchant wallet. Format +964XXXXXXXXXX.
    store_password Shared secret bound to your store. Never share it with customers and never embed it in client-side code or mobile binaries.

    These values are sent in the request body of server-to-server calls. They are validated as a paired credential — the gateway will not tell you whether the failure was the mobile or the password (this prevents account enumeration).

    2.2 Where these credentials are accepted¶

    Endpoint merchant_mobile_no store_password
    POST /merchant/generate-payment-token required required
    POST /merchant/generate-qr-token required required
    POST /merchant/validate-payment required required
    POST /merchant/fetch-payment-status required n/a (order ID + amount serve as proof of knowledge)
    POST /merchant/fetch-payment-token-data n/a n/a (read-only, token is the lookup key)
    POST /merchant/cash-back required required
    POST /merchant/payment/direct n/a (customer credentials, not merchant) n/a

    2.3 Network controls¶

    Two controls are available beyond the credential pair. Both are configured from the merchant dashboard and are strongly recommended for production:

    1. Source IP allow-list — the gateway will reject server-to-server calls that arrive from any IP outside the list you provision. Use this if your platform issues outbound calls from a stable egress range.
    2. TLS enforcement — HTTPS is the only accepted scheme. There is no HTTP listener to fall back to.

    2.4 IPN (callback) authenticity¶

    The gateway delivers payment-outcome callbacks to the ipn_url you configure. The default callback is a application/x-www-form-urlencoded POST. To prove the callback originated from ZiCharge, two layers are available:

    Layer 1 — connection-level. Configure your firewall to accept POSTs to your IPN listener only from the ZiCharge production egress range published in the merchant dashboard. This is the simplest defence and is sufficient for most merchants.

    Layer 2 — payload signature (recommended for new integrations). When enabled on your merchant configuration, every IPN includes:

    • X-ZiCharge-Signature HTTP header — hex-encoded HMAC-SHA256 of the raw form body, signed with your ipn_secret.
    • X-ZiCharge-Timestamp HTTP header — Unix epoch seconds at dispatch time.
    • X-ZiCharge-Request-Id HTTP header — opaque request ID, useful for correlating with your logs and the gateway's logs.

    To verify:

    import hmac, hashlib
expected = hmac.new(
    ipn_secret.encode("utf-8"),
    raw_body_bytes,
    hashlib.sha256
).hexdigest()
if not hmac.compare_digest(expected, received_signature):
    return reject()
# Also enforce: abs(now - X-ZiCharge-Timestamp) < 300 seconds

    Implementation notes:

    • Compare signatures in constant time (hmac.compare_digest, MessageDigest.isEqual) to prevent timing attacks.
    • Reject any IPN where the timestamp drifts more than ±5 minutes from your server clock.
    • Rotate the ipn_secret from the merchant dashboard; the gateway supports two active secrets during rotation windows.

    If your merchant configuration does not have payload signatures enabled yet, contact integrations support to switch it on. Existing integrations continue to work with Layer 1 only.

    2.5 Idempotency¶

    The gateway is idempotent on the inputs that matter:

    • Token mint — calling generate-payment-token repeatedly with the same merchant_mobile_no + order_id while a token is still valid will return the existing token; it will not create a duplicate.
    • Direct payment — protected by the token's lifecycle state machine. The same payment_token cannot be charged twice; the second attempt fails with a token_already_used error.
    • IPN delivery — every IPN we send carries an X-ZiCharge-Request-Id header. We may re-deliver after a transient error; deduplicate on your side using this ID plus order_id.

    2.6 Replay protection¶

    For server-to-server calls, idempotency keys plus the short token validity window (15 minutes) bound replay risk. For IPN traffic, the X-ZiCharge-Timestamp + signature pair provides explicit replay protection.

    2.7 Sensitive data handling¶

    • Never log store_password, payment_token, or customer credentials.
    • Never include store_password or payment_token in URLs (use POST body).
    • Never expose your store_password to a customer browser; all credentialed calls are server-to-server only.
    • The gateway redacts sensitive fields in its own logs as a defence in depth; that is not a substitute for your own discipline.

    2.8 Internal endpoint protection¶

    A small set of /internal/merchant/* endpoints exist for scheduler use (cancel-IPN retry, reconciliation). These are not part of the public surface, are protected by a separate X-Internal-Auth shared secret compared in constant time, and are additionally restricted at the load balancer / ingress layer. They are documented here only so that you do not waste time discovering them in network traces.
