    3. Payment Flow¶

    This chapter walks through the end-to-end checkout from the moment your customer presses "Pay" until your order is fulfilled. Read this before implementing — the API reference will then read as a glossary rather than a puzzle.

    3.1 Sequence — happy path (hosted page)¶

    Customer        Your Backend                ZiCharge Gateway              Your IPN Listener
   │                │                              │                                │
   │  add to cart   │                              │                                │
   │ ──────────────►│                              │                                │
   │                │                              │                                │
   │                │ POST /merchant/generate-     │                                │
   │                │      payment-token           │                                │
   │                │ ────────────────────────────►│                                │
   │                │                              │  mint token (15-min validity)  │
   │                │◄──────────────────────────── │  { code:200, token:"..." }     │
   │                │                              │                                │
   │  HTTP 302 to   │                              │                                │
   │  ZiCharge      │                              │                                │
   │◄───────────────│                              │                                │
   │ GET /merchant/payment?token=...               │                                │
   │ ─────────────────────────────────────────────►│                                │
   │  hosted page (mobile + password / PIN)        │                                │
   │◄───────────────────────────────────────────── │                                │
   │                │                              │                                │
   │  submit credentials                           │                                │
   │ ─────────────────────────────────────────────►│  authenticate, debit, credit   │
   │                │                              │  mark token USED               │
   │  success screen + auto-redirect to            │                                │
   │  merchant success_url                         │                                │
   │◄───────────────────────────────────────────── │                                │
   │                │                              │                                │
   │                │                              │  AFTER_COMMIT, async:          │
   │                │                              │  POST ipn_url (form-encoded)   │
   │                │                              │ ──────────────────────────────►│
   │                │                              │                                │  verify signature,
   │                │                              │                                │  store transaction,
   │                │                              │                                │  HTTP 200 OK
   │                │                              │◄────────────────────────────── │
   │                │                              │                                │
   │                │  POST /merchant/             │                                │
   │                │       fetch-payment-status   │                                │
   │                │ ────────────────────────────►│                                │
   │                │  { code:200, paid:true,      │                                │
   │                │    message:"Successfully     │                                │
   │                │    Deposited." }             │                                │
   │                │◄──────────────────────────── │                                │
   │                │                              │                                │
   │                │  fulfil order                │                                │

    Two confirmation channels — the IPN and the synchronous fetch-payment-status / validate-payment — are deliberate. Either one is authoritative on its own; both together are how a robust integration confirms before fulfilling.

    3.2 Sequence — customer cancels or times out¶

    Customer        Your Backend                ZiCharge Gateway              Your IPN Listener
   │                │                              │                                │
   │  on hosted page, clicks Cancel ✕              │                                │
   │  (or 15-min timer hits 0)                     │                                │
   │ ─────────────────────────────────────────────►│  mark token CANCELLED          │
   │                │                              │  (first-write-wins lock)       │
   │  "Payment cancelled" → redirect to            │                                │
   │  merchant cancel_url                          │                                │
   │◄───────────────────────────────────────────── │                                │
   │                │                              │                                │
   │                │                              │  AFTER_COMMIT, async:          │
   │                │                              │  POST ipn_url                  │
   │                │                              │   status=Cancelled             │
   │                │                              │ ──────────────────────────────►│
   │                │                              │  retry policy (see §3.4)       │

    3.3 State machine — payment token¶

                      ┌───────────┐
                  │  PENDING  │  token minted, not paid, not cancelled, within validity window
                  └─────┬─────┘
                        │
        ┌───────────────┼────────────────────┬────────────────────┐
        │               │                    │                    │
        ▼               ▼                    ▼                    ▼
   Customer pays   Customer cancels    Timer expires while   Customer opens an
                   on hosted page      page is open          already-expired link
        │               │                    │                    │
        ▼               ▼                    ▼                    ▼
   ┌────────┐     ┌──────────┐         ┌──────────┐         ┌──────────┐
   │  USED  │     │CANCELLED │         │CANCELLED │         │CANCELLED │
   │ (paid) │     │ by USER  │         │ by USER  │         │by SYSTEM │
   └────────┘     │USER_CAN- │         │TIMEOUT_  │         │TIMEOUT_  │
                  │ CELLED   │         │CLIENT    │         │DIRECT_   │
                  └──────────┘         └──────────┘         │OPEN      │
                                                            └──────────┘
                       │                    │                    │
                       └────────────────────┴────────────────────┘
                                            │
                                IPN dispatched (Success | Cancelled)

    Transitions are guarded by:

    • A pessimistic WRITE lock on the token row during direct-payment processing (3-second wait timeout — fail-fast under contention).
    • A conditional UPDATE on cancel (WHERE cancelled_at IS NULL) — first-write-wins, no silent overwrites.
    • Idempotent dispatch claim before each IPN send — duplicate IPNs are physically impossible across overlapping retry runs.

    3.4 IPN delivery & retries¶

    When a terminal state is reached, the gateway publishes a domain event. After the database transaction commits, an asynchronous listener POSTs the IPN to your ipn_url.

    Delivery semantics:

    • At-least-once. You may receive the same IPN more than once; deduplicate using order_id + X-ZiCharge-Request-Id.
    • Order: for a single token, you will never receive a Cancelled IPN after a Success IPN. The state machine prevents both terminal states.
    • Timing: IPNs are dispatched on the gateway's asynchronous worker pool, typically within a few hundred milliseconds of commit. Treat anything under 5 seconds as nominal.

    Retry policy for failed deliveries (your listener was unreachable, returned non-2xx, or timed out):

    Attempt When
    1 Immediately after commit
    2..N Driven by an internal scheduler with a per-token cooldown (default several minutes)
    Max 8 attempts per terminal event

    Between attempts the gateway records:

    • ipn_cancel_attempts (or success-side equivalent) — counter, monotonic.
    • ipn_cancel_last_attempt_at — last dispatch timestamp; used by the cooldown gate to prevent duplicate retries across overlapping scheduler runs.
    • ipn_cancel_fired_at — set only after a 2xx confirmation; absence indicates "not yet successfully delivered".
    • The last error string (class + truncated message, max 500 chars) for support diagnosis.

    After 8 failed attempts the token stops being picked up. Your reconciliation flow should use POST /merchant/fetch-payment-status on order timeout to recover.

    What your listener must do for retries to behave well:

    • Respond with HTTP 2xx only after you have persisted the IPN. A 2xx response is taken as proof of delivery — we will not retry.
    • If you cannot persist yet (database unavailable), respond with HTTP 5xx. We will retry.
    • Time-limit your own processing — slow listeners (> 10 seconds) may be considered failed and retried.

    3.5 Direct payment flow¶

    For merchants integrating their own customer-facing checkout (mobile app, in-store POS, custom web flow), the direct payment endpoint accepts the customer's credentials and the payment_token in a single server-to-server call:

    POST /merchant/payment/direct
{
  "mobile_no":     "+9647701234567",
  "password":      "<customer-password>",
  "payment_token": "<token-from-step-1>"
}

    The flow is otherwise identical — token transitions to USED, IPN is fired, the success envelope is returned synchronously.

    Direct payment requires that your channel be subject to PCI-equivalent controls on handling customer passwords. Specifically:

    • Never persist the customer's password.
    • Use a dedicated transport that you control end-to-end.
    • Prefer the hosted page for any web-browser context.

    3.6 Reconciliation¶

    Reconciliation has two server-to-server endpoints. Use them when the IPN is late, lost, or your listener was being deployed:

    Endpoint Use it when
    POST /merchant/fetch-payment-status You know the order_id and the amount — fast, exact match
    POST /merchant/validate-payment You want the full transaction detail (timestamp, customer account, transaction id)

    Both endpoints are read-only and safe to call as often as needed.

    3.7 Timing — what to expect¶

    Operation Typical P99 budget
    generate-payment-token < 100 ms 300 ms
    validate-payment / fetch-payment-status < 80 ms 250 ms
    payment/direct end-to-end (synchronous portion) < 500 ms 1.5 s
    Token validity window n/a 15 minutes from mint
    IPN dispatch after commit < 1 s nominal retries up to 8 over a multi-minute window

    These are gateway-side budgets and exclude your own backend latency. Configure your HTTP clients with explicit connect / read timeouts (suggest 2 s connect, 10 s read).
