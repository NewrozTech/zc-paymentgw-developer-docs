    4. API Reference¶

    All endpoints return HTTP 200; the logical outcome is in the body code field. Paths shown without prefix are mounted under both /api/v3/merchant and /merchant.

    Base URLs:

    • Sandbox — https://dev.zicharge.com
    • Production — https://secure.zicharge.com

    POST /merchant/generate-payment-token¶

    Mint a one-time payment token for a checkout. Idempotent per (merchant_mobile_no, order_id) while a prior token is still valid.

    Request — JSON¶

    POST /merchant/generate-payment-token HTTP/1.1
Host: secure.zicharge.com
Content-Type: application/json

    {
  "merchant_mobile_no": "+9647712345678",
  "store_password":     "your-store-secret",
  "order_id":           "ORD-2026-000123",
  "bill_amount":        5000,
  "success_url":        "https://shop.example.com/orders/123/success",
  "cancel_url":         "https://shop.example.com/orders/123/cancel",
  "fail_url":           "https://shop.example.com/orders/123/fail",
  "customer_mobile_no": "+9647701234567"
}

    Request — form-urlencoded¶

    The same fields as the JSON body are accepted as form fields. Use this only for parity with older integrations.

    Field reference¶

    Field Type Required Notes
    merchant_mobile_no string yes +964XXXXXXXXXX, validated by Iraqi mobile-number regex
    store_password string yes Shared secret from the merchant dashboard
    order_id string yes Your unique order identifier; idempotency anchor
    bill_amount number yes IQD; minimum 250; whole numbers recommended
    success_url URL no Customer redirected here after payment; falls back to merchant default
    cancel_url URL no Customer redirected here on cancel; falls back to merchant default
    fail_url URL no Customer redirected here on failure; falls back to merchant default
    customer_mobile_no string no Pre-fill the customer's mobile on the hosted page; format +964XXXXXXXXXX if present

    Success response (legacy envelope, preserved for client compatibility)¶

    {
  "messages": [],
  "token":    "tkn_4f8a6e1fa8df48e88ef49a8sd7f458ve4s4dfgpkk",
  "code":     200
}

    Redirect your customer to:

    https://secure.zicharge.com/merchant/payment?token=<token>

    Common error responses¶

    code When
    422 Validation failure — missing field, malformed mobile, sub-minimum amount; messages contains one entry per failed field
    401 Invalid merchant_mobile_no / store_password pair
    409 Order already paid by a previous token
    500 Unexpected — log the X-ZiCharge-Request-Id response header and contact support

    ℹ️ The canonical, non-legacy path POST /merchant/generate-qr-token returns the same token plus a server-generated qr_url and expires_in_minutes. See below.

    POST /merchant/generate-qr-token¶

    Mints a token and additionally returns a QR-code URL pre-encoded with the payment link. Use this for in-store POS terminals, kiosks, and printed media. The token semantics are otherwise identical to generate-payment-token.

    Request¶

    Same fields as generate-payment-token.

    Success response¶

    {
  "code": 200,
  "messages": ["ok"],
  "data": {
    "token":              "tkn_4f8a6e1fa8df48e88ef49a8sd7f458ve4s4dfgpkk",
    "qr_url":             "https://secure.zicharge.com/qr/tkn_4f8a6e1fa8df48e88ef49a8sd7f458ve4s4dfgpkk.png",
    "expires_in_minutes": 15
  }
}

    POST /merchant/validate-payment¶

    Authoritative server-to-server lookup for the result of a completed order. Use this on the reconciliation path or as a synchronous confirm before fulfilling.

    Aliases: the path POST /merchant/payment/validation is identical and is preserved for compatibility.

    Request¶

    {
  "merchant_mobile_no": "+9647712345678",
  "store_password":     "your-store-secret",
  "order_id":           "ORD-2026-000123"
}

    Success response¶

    {
  "code": 200,
  "messages": ["Please find your transaction details:"],
  "data": {
    "transaction_id":     123456,
    "order_id":           "ORD-2026-000123",
    "bill_amount":        "5000",
    "customer_account_no":"+9647701234567",
    "status":             "Success",
    "received_at":        "2026-05-14 12:55:45"
  }
}

    Errors¶

    code When
    422 Validation failure (missing / malformed field)
    422 Credential pair invalid OR order not found — single error to prevent enumeration. messages contains "Sorry! Invalid Store Credentials." or "Sorry! Invalid Order ID."

    POST /merchant/fetch-payment-status¶

    Fast yes/no check for a specific (order, amount) pair. Cheaper than validate-payment because the lookup is an exact-match on the indexed columns.

    Also available as GET /merchant/fetch-payment-status with the same parameters in the query string. The GET form is convenient for retries from systems that prefer idempotent verbs.

    Request¶

    {
  "merchant_mobile_no": "+9647712345678",
  "order_id":           "ORD-2026-000123",
  "amount":             5000
}

    Success response¶

    {
  "code": 200,
  "messages": ["Congratulation! Successfully Deposited."],
  "data": {
    "paid":    true,
    "message": "Congratulation! Successfully Deposited."
  }
}

    Non-success response (order not yet paid)¶

    {
  "code": 404,
  "messages": ["Awaiting payment for order id #ORD-2026-000123"],
  "data": {
    "paid":    false,
    "message": "Awaiting payment for order id #ORD-2026-000123"
  }
}

    HTTP status remains 200; only the body code differs. The exact match is performed at the database level on (merchant_id, order_id, amount) for deterministic behaviour.

    POST /merchant/fetch-payment-token-data¶

    Returns the public-safe metadata for a token, by token. Used by the hosted payment page; also available to merchants who want to inspect a token's state without performing a validate-payment lookup.

    Also available as GET /merchant/fetch-payment-token-data?payment_token=... with CORS enabled for browser-side inspection.

    Request¶

    { "payment_token": "tkn_..." }

    Success response¶

    {
  "code": 200,
  "messages": ["ok"],
  "data": {
    "token":              "tkn_4f8a6e1fa8df48e88ef49a8sd7f458ve4s4dfgpkk",
    "merchant_mobile_no": "+9647712345678",
    "merchant_name":      "Example Store",
    "order_id":           "ORD-2026-000123",
    "bill_amount":        "5000",
    "success_url":        "https://shop.example.com/orders/123/success",
    "cancel_url":         "https://shop.example.com/orders/123/cancel",
    "fail_url":           "https://shop.example.com/orders/123/fail",
    "customer_mobile_no": "+9647701234567",
    "status":             "PENDING",
    "created_at":         "2026-05-14T12:50:00"
  }
}

    status is one of PENDING, USED, CANCELLED.

    POST /merchant/payment/direct¶

    Charge a payment token directly using customer credentials. Server-to-server call from a trusted channel only.

    Request¶

    {
  "mobile_no":     "+9647701234567",
  "password":      "<customer-password>",
  "payment_token": "tkn_4f8a6e1fa8df48e88ef49a8sd7f458ve4s4dfgpkk"
}

    Success response (legacy envelope)¶

    {
  "messages": ["Payment processed successfully."],
  "transactionInfo": {
    "id":                  123456,
    "order_id":            "ORD-2026-000123",
    "bill_amount":         5000,
    "customer_account_no": "+9647701234567",
    "status":              "Success",
    "received_at":         "2026-05-14 12:55:45"
  },
  "code": 200
}

    The IPN to your ipn_url is fired asynchronously after commit, identical to the hosted-page flow.

    Errors¶

    code When
    422 Validation failure
    422 "Mobile No. & Password combination do not match." — unified message; intentionally indistinguishable from "blocked account" to prevent enumeration
    404 Token unknown
    410 Token expired
    409 Token already used
    409 Token cancelled
    422 Insufficient balance, limit exceeded, KYC tier insufficient — exact message in messages[0]

    GET /merchant/transactions¶

    Paginated list of completed transactions for a merchant. Read-only.

    Query parameters¶

    Name Type Default Notes
    merchantMobileNo string required Your merchant mobile number
    page int 1 1-indexed page number
    perPage int 20 Cap 100

    Response¶

    {
  "code": 200,
  "messages": ["ok"],
  "data": {
    "transactions": [
      {
        "transaction_id":     123456,
        "order_id":           "ORD-2026-000123",
        "bill_amount":        "5000",
        "customer_account_no":"+9647701234567",
        "status":             "Success",
        "received_at":        "2026-05-14 12:55:45"
      }
    ],
    "current_page": 1,
    "per_page":     20,
    "total":        4231,
    "message":      "ok"
  }
}

    POST /merchant/cash-back¶

    Push a refund or promotional credit from your merchant wallet to a customer wallet. Settled synchronously.

    Request¶

    {
  "merchant_mobile_no": "+9647712345678",
  "store_password":     "your-store-secret",
  "receiver_mobile_no": "+9647701234567",
  "amount":             1500,
  "reference_id":       "RFD-2026-000789",
  "lang":               "en"
}

    Field reference¶

    Field Type Required Notes
    merchant_mobile_no string yes Your merchant wallet
    store_password string yes
    receiver_mobile_no string yes Must be an existing customer wallet
    amount number yes IQD, minimum 250
    reference_id string no Your reference; idempotency anchor
    lang string no en, ar, ku — overrides Accept-Language

    Response¶

    {
  "code": 200,
  "messages": ["Cash-back processed successfully."],
  "data": {
    "transaction_id":      234567,
    "reference_id":        "RFD-2026-000789",
    "amount":              1500,
    "customer_account_no": "+9647701234567",
    "status":              "Success",
    "sent_at":             "2026-05-14 13:02:11"
  }
}

    Errors¶

    Standard envelope. Notable cases: insufficient merchant balance (code 422), receiver not found (code 422 generic), daily cap exceeded (code 422).

    Hosted Payment Page¶

    GET /merchant/payment?token=<token>¶

    Returns the HTML payment page for the customer. Not an API endpoint per se — redirect your customer here. The page is mobile-responsive, supports iframe embedding via ?iframe=1, and localizes itself based on Accept-Language.

    If the token is missing, expired, or already settled, the page renders an explanatory message and (for expired tokens) triggers a server-side cancel that will dispatch a Cancelled IPN.

    Common request headers¶

    Header Notes
    Content-Type application/json (recommended) or application/x-www-form-urlencoded
    Accept-Language en, ar, ku — affects message localization
    X-Request-Id Optional client-supplied request ID; if absent, the gateway assigns one and returns it in X-ZiCharge-Request-Id

    Common response headers¶

    Header Notes
    X-ZiCharge-Request-Id Echoed from X-Request-Id or freshly assigned. Include this in support requests.
    Content-Type Always application/json; charset=utf-8 on API endpoints
