    5. IPN (Instant Payment Notification) Callbacks¶

    IPN is the gateway's asynchronous notification to your platform when a payment reaches a terminal state (Success or Cancelled). It is the same mechanism your existing integration already uses; this chapter documents the contract, the retry behaviour, and the verification steps in full.

    5.1 Contract¶

    The gateway POSTs to the ipn_url configured for your merchant account.

    • Method: POST
    • Content-Type: application/x-www-form-urlencoded
    • Body encoding: standard form encoding, UTF-8

    Payload fields¶

    Field Type Success value Cancelled value
    transaction_id string numeric ledger transaction id empty string ("")
    order_id string the order_id you submitted at token mint same
    bill_amount string decimal, e.g. "5000" decimal, e.g. "5000"
    customer_account_no string customer's mobile, e.g. "+9647701234567" customer's mobile if known, else empty
    status string "Success" "Cancelled"
    received_at string "yyyy-MM-dd HH:mm:ss" (gateway clock, UTC) same
    ipn string "1" "1"

    Headers (when signature verification is enabled on your config)¶

    Header Description
    X-ZiCharge-Signature Hex HMAC-SHA256 of the raw form body, signed with your ipn_secret
    X-ZiCharge-Timestamp Unix epoch seconds at dispatch time
    X-ZiCharge-Request-Id Opaque request ID; use to deduplicate and to correlate with support

    5.2 Verification — recommended listener pseudocode¶

    def handle_ipn(request):
    body = request.raw_body  # bytes — verify against the raw body, not parsed form

    # 1) Signature
    expected = hmac.new(IPN_SECRET, body, hashlib.sha256).hexdigest()
    signature = request.headers["X-ZiCharge-Signature"]
    if not hmac.compare_digest(expected, signature):
        return Response(status=400)

    # 2) Timestamp drift (±5 minutes)
    ts = int(request.headers["X-ZiCharge-Timestamp"])
    if abs(time.time() - ts) > 300:
        return Response(status=400)

    # 3) Idempotency
    request_id = request.headers["X-ZiCharge-Request-Id"]
    if already_processed(request_id):
        return Response(status=200)   # we already stored it — acknowledge

    # 4) Parse & persist
    fields = parse_form(body)
    assert fields["status"] in ("Success", "Cancelled")
    persist_ipn(fields, request_id)

    # 5) Acknowledge ONLY after persistence
    return Response(status=200)

    Three pieces of discipline matter:

    1. Sign-verify against the raw body, not the parsed form. A re-encoded form is not byte-identical and the HMAC will not match.
    2. Constant-time signature comparison (hmac.compare_digest, MessageDigest.isEqual) to avoid timing leakage.
    3. Acknowledge only after persistence. A 2xx response stops retries. If you cannot persist, return 5xx and we will retry.

    5.3 Delivery semantics¶

    • At-least-once delivery. Build your listener to be idempotent on X-ZiCharge-Request-Id (preferred) or on order_id + status.
    • No Success after Cancelled and no Cancelled after Success. The token state machine guarantees one terminal outcome per token.
    • Cooldown-gated retries. Concurrent retry workers cannot dispatch the same IPN twice; this is enforced by an atomic claim before each send.
    • Maximum 8 attempts per terminal event. After that, recovery shifts to your synchronous reconciliation calls (fetch-payment-status / validate-payment).

    5.4 What "delivery succeeded" means¶

    The gateway records a successful delivery when your listener returns any HTTP 2xx response within the read timeout. The response body is discarded; only the status code matters.

    • HTTP 2xx → delivered, no more retries for this event.
    • HTTP 3xx → not followed; treated as failed; retry scheduled.
    • HTTP 4xx → treated as failed; retry scheduled. Do not respond 4xx unless you genuinely want the gateway to retry — typically this only delays your own processing.
    • HTTP 5xx → failed; retry scheduled.
    • Connection refused / read timeout (~10 s) → failed; retry scheduled.

    5.5 SSRF protection on the gateway side¶

    The gateway will refuse to POST your IPN to a URL that resolves to an internal-only address range. Configure your ipn_url to a publicly resolvable HTTPS endpoint. URLs that resolve to RFC 1918 ranges, link-local, loopback, or other reserved blocks are rejected and the IPN is dropped (with a structured log on the gateway side).

    5.6 Configuring your IPN URL¶

    Set the ipn_url on your merchant configuration through the dashboard. Recommended:

    • HTTPS only.
    • A dedicated path (e.g., /internal/zicharge/ipn) protected by your own ingress.
    • Optional: rotate the ipn_secret from the dashboard. The gateway accepts two active secrets during rotation windows so you can switch over without downtime.

    5.7 Observability¶

    For every IPN you receive, the gateway has emitted structured events with the same X-ZiCharge-Request-Id. If a delivery is in dispute, supply that header value to support and we can show you the dispatch attempts, response codes, retry timings, and (if signatures were enabled) the bytes signed.

    5.8 Synchronous fallback¶

    If your listener has been down longer than the retry window allows, recover with:

    • POST /merchant/fetch-payment-status — fast, returns the boolean outcome.
    • POST /merchant/validate-payment — full detail.

    Both endpoints are safe to call repeatedly.
