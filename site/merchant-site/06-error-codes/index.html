    6. Error Codes¶

    Every API endpoint returns HTTP 200; the logical outcome is in the body code field. The catalogue below covers every value you should expect to see in production, and the recommended client handling for each.

    6.1 Success¶

    code Meaning Recommended action
    200 Request fulfilled. data contains the endpoint payload. Proceed.

    6.2 Client errors¶

    code Meaning Where it appears Recommended action
    400 Malformed request body / unsupported content type All endpoints Fix the request; do not retry as-is.
    401 Authentication required or invalid (merchant credentials) All credentialed endpoints Verify merchant_mobile_no + store_password; do not auto-retry — exponential lockout protects accounts.
    404 Resource not found in this context fetch-payment-status, fetch-payment-token-data Treat as "not yet paid / unknown token". For status checks, retry per your reconciliation policy.
    409 Conflict — state machine refused the transition Token mint (duplicate order), direct-payment (token already used / cancelled) Do not retry. Inspect with fetch-payment-token-data if needed.
    410 Gone — token expired Direct payment, hosted page Mint a new token.
    422 Validation failure or business-rule violation All endpoints Read messages[] — each entry corresponds to one rule violation. Fix and retry.
    429 Rate-limited Any credentialed endpoint under sustained abuse Back off exponentially. Contact integrations support if you legitimately exceed rate limits.

    6.3 Server errors¶

    code Meaning Recommended action
    500 Unexpected gateway error Retry idempotent reads (fetch-payment-status, validate-payment) with backoff. For writes (generate-payment-token, payment/direct), verify with fetch-payment-status / fetch-payment-token-data before retrying. Include X-ZiCharge-Request-Id in support tickets.
    502 / 503 / 504 Transient infrastructure failure (rare; expected only during deploys) Retry with exponential backoff (e.g., 1 s, 2 s, 4 s, up to 30 s total).

    6.4 Common messages strings¶

    Localized; the strings below are the canonical English forms.

    String Trigger
    "Sorry! Invalid Store Credentials." Merchant credential pair did not match. Intentionally generic to prevent enumeration.
    "Sorry! Invalid Order ID." Credentials valid, but the order was not found.
    "Mobile No. & Password combination do not match." Direct-payment customer credentials invalid or account blocked. Intentionally generic.
    "Congratulation! Successfully Deposited." Successful payment confirmed by fetch-payment-status.
    "Awaiting payment for order id #<order_id>" fetch-payment-status did not find a completed match.
    "The merchant mobile no format is invalid." Phone-number regex failure on merchant_mobile_no.
    "The customer mobile no format is invalid." Phone-number regex failure on customer_mobile_no.
    "Bill amount must be at least 250." Sub-minimum amount at token generation.
    "This payment link has expired." Hosted page accessed after the 15-minute validity window.
    "Insufficient balance." Direct payment with under-funded customer wallet.
    "Daily limit exceeded." / "Monthly limit exceeded." Customer or merchant transaction cap reached.
    "Unable to process service!" Generic fallback for unexpected server errors.

    6.5 Validation envelope¶

    When a request fails Bean Validation (@NotBlank, regex, @NotNull, etc.), the response contains one entry per failed field:

    {
  "code": 422,
  "messages": [
    "The merchant mobile no format is invalid.",
    "The order id must not be blank."
  ],
  "data": null
}

    The messages preserve the same wording your previous integration would have seen, so any client-side string matching you may have stays valid.

    6.6 Retry guidance — summary¶

    Operation Idempotent? Retry on 500/5xx?
    generate-payment-token Yes per (merchant, order_id) while a prior token is still valid Yes — same order_id returns the same token
    validate-payment Yes (read) Yes
    fetch-payment-status Yes (read) Yes
    fetch-payment-token-data Yes (read) Yes
    payment/direct Yes — the token's state machine prevents double-charge Yes, but verify with fetch-payment-token-data first
    cash-back Yes per reference_id Yes — same reference_id returns the same result
    transactions (list) Yes (read) Yes
