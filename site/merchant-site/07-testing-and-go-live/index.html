    7. Testing & Go-Live¶

    7.1 Sandbox¶
    Base URL https://dev.zicharge.com
    Purpose Functional integration testing, no real funds movement
    Lifetime Persistent; data is retained but not warranted
    Throughput Generous, but not load-tested — do not stress-test sandbox
    Test wallets Provisioned on request from integrations support

    Sandbox shares the same wire contract as production. The only difference is that all settlement is simulated.

    7.2 Test plan checklist¶

    Run this list against sandbox before requesting production credentials.

    Functional¶

    • Mint a token with the canonical JSON request and the form-urlencoded request — verify both succeed.
    • Customer pays on the hosted page — verify the IPN arrives and fetch-payment-status returns paid: true.
    • Customer cancels on the hosted page — verify the cancel IPN arrives with status=Cancelled.
    • Let a token expire (15 min) — verify no IPN fires while it is unopened, and that opening it after expiry triggers a system cancel + cancel IPN.
    • Direct payment with a valid token — verify success envelope and IPN.
    • Direct payment with an already-used token — verify code 409.
    • Direct payment with an expired token — verify code 410.
    • fetch-payment-status for an unpaid order — verify paid: false and code 404 in the body.
    • fetch-payment-status for a paid order — verify paid: true and code 200.
    • validate-payment for a paid order — verify the full detail payload.
    • Cash-back to a known wallet — verify success.
    • Cash-back retry with the same reference_id — verify the same result is returned (idempotency).
    • List transactions, page through more than one page.

    Negative¶

    • Wrong store_password — code 401, generic message.
    • Malformed merchant_mobile_no — code 422.
    • bill_amount < 250 — code 422.
    • Missing required field — code 422, message names the field.
    • Direct-payment with a bad customer password — code 422, unified message.

    IPN listener¶

    • Listener returns 2xx only after persistence.
    • Listener handles repeated delivery of the same X-ZiCharge-Request-Id and does not double-process.
    • If signatures are enabled: signature mismatch is rejected; valid signature is accepted.
    • Timestamp drift > 5 minutes is rejected.
    • Listener survives a brief outage and accepts the eventual retry.

    Resilience¶

    • Inject a 500 from your listener — verify the gateway retries within the documented window.
    • After listener recovery, verify the retry succeeds and no duplicate is stored on your side.

    Reconciliation¶

    • Disable your IPN listener entirely, complete a payment, then run fetch-payment-status and verify you can recover the outcome.

    7.3 Go-live checklist¶

    When the sandbox plan is green:

    • Production merchant_mobile_no and store_password provisioned and stored in your secrets manager — never in source control.
    • Production ipn_url configured. HTTPS, public, your own ingress.
    • (Recommended) IPN signature verification enabled; ipn_secret stored in secrets manager.
    • (Recommended) Source IP allow-list configured for outbound server-to-server calls.
    • Your egress IPs registered with integrations support if you intend to allow-list at our edge.
    • Reconciliation cron configured for orders that pass a timeout without an IPN.
    • Logging: you record X-ZiCharge-Request-Id for every gateway interaction (outbound and IPN).
    • Alerting: at minimum, a non-2xx rate alert on your IPN listener and a "no IPN within N minutes after token mint" alert on your order pipeline.

    7.4 Operational expectations¶

    • Uptime target: 99.9% monthly on the production gateway.
    • Maintenance windows: announced in advance through the integrations channel.
    • Deploys: immutable, rolling — no downtime in the steady state. A small number of in-flight requests may receive a 5xx during instance shift; retries on the idempotent endpoints recover transparently.
    • P99 latency targets are documented in 03-payment-flow.md §3.7.

    7.5 Credential rotation¶

    • store_password — rotate from the merchant dashboard. The gateway supports a brief overlap window during rotation so you can deploy the new value with zero downtime.
    • ipn_secret — same rotation model: two active secrets supported during rotation.
    • Customer credentials — never handled by you in any form; customers rotate their own from the ZiCharge app.

    7.6 Support¶

    • Integration questions: integrations@zicharge.com (placeholder; replace with your actual channel).
    • Production incidents: the support escalation path documented in your merchant agreement.
    • Always include: X-ZiCharge-Request-Id, order_id, approximate timestamp, environment (sandbox / production).
