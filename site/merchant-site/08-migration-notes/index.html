    8. Migration Notes — Previous Gateway vs. Current Java Engine¶

    TL;DR for existing merchants: Your current integration keeps working. Endpoint URLs, request shapes, response shapes, and the IPN payload are preserved. There are seven small behavioural differences listed below — none are blockers if you follow the recommendations. New, optional capabilities (signed IPN, QR token endpoint, idempotency on cash-back) are additive.

    The current gateway is a complete re-engineering of the merchant payment surface on a Java 21 / Spring Boot 3.2 backend. The re-engineering was scoped explicitly to preserve the journey your merchants already know while strengthening validation, idempotency, race-safety, and observability under the hood.

    This chapter is the comparison map. Read it once during your upgrade audit, then keep it for reference.

    8.1 What is unchanged (and warranted to stay so)¶

    These items are part of the wire contract and are guaranteed within this major version.

    Surface Status
    Endpoint paths — POST /merchant/generate-payment-token, POST /merchant/payment/validation, POST /merchant/payment/direct, the hosted GET /merchant/payment?token=... ✅ Same
    Equivalent canonical paths under /api/v3/merchant/... are also available (recommended for new code) ✅ Same behaviour
    Request field names — merchant_mobile_no, store_password, order_id, bill_amount, success_url, cancel_url, fail_url, customer_mobile_no, payment_token, mobile_no, password ✅ Same
    Credential model — merchant_mobile_no + store_password server-to-server, hosted page for customer credentials ✅ Same
    Hosted-page URL — https://<host>/merchant/payment?token=<token> ✅ Same
    Token validity window — 15 minutes from mint ✅ Same
    Response envelope — { messages, data, code }, HTTP 200 for all outcomes ✅ Same
    Legacy success envelope for generate-payment-token — { messages, token, code } ✅ Same
    Legacy success envelope for payment/direct — { messages, transactionInfo, code } ✅ Same
    IPN method, content-type, and field set — POST application/x-www-form-urlencoded with transaction_id, order_id, bill_amount, customer_account_no, status, received_at, ipn=1 ✅ Same
    IPN status values — Success, Cancelled ✅ Same
    Both JSON and form-urlencoded request bodies accepted on server-to-server endpoints ✅ Same

    If your existing integration only uses the items above, it requires no changes.

    8.2 Behavioural differences you should know about¶

    Each item below lists: the difference, the rationale, the impact on a live integration, and the recommended action.

    Δ-1 — Iraqi mobile-number regex is now enforced on payment endpoints¶
    What changed merchant_mobile_no, customer_mobile_no (when present), and customer mobile_no on direct payment are validated against ^\+964(0)?[0-9]{9,10}$.
    Why The previous gateway also enforced this regex on the same endpoints. The Java engine was briefly more lax during the rebuild; this restores parity.
    Impact on you If your client was submitting numbers without the +964 prefix, they were failing downstream anyway. They now fail at validation with a clear 422.
    Action Send all mobiles in +964XXXXXXXXXX format. Validation message: "The merchant mobile no format is invalid." / "The customer mobile no format is invalid.".
    Blocker? ❌ No. Same behaviour as the previous gateway.

    Note (intentional carve-out): Cash-back (POST /merchant/cash-back) does not enforce this regex on receiver_mobile_no, matching the previous gateway. The receiver is validated by existence in the user table.

    Δ-2 — Minimum bill_amount enforced at token generation¶
    What changed bill_amount < 250 is rejected at token mint with code 422, instead of being accepted and failing later at payment time.
    Why Fail-fast surfaces the error to your client at the moment your server has the full request context, not in the middle of a customer-facing checkout.
    Impact on you If you previously minted sub-minimum tokens for testing, you must now use ≥ 250. Production traffic that already succeeded at payment time is unaffected.
    Action Ensure your minimum order value is ≥ 250 IQD before calling token mint.
    Blocker? ❌ No. The previous behaviour was a bug; the engineered behaviour is the documented limit.

    Δ-3 — Direct-payment error message is unified¶
    What changed When the customer's credentials are invalid for any reason (unknown user, wrong password, blocked account), the message returned is the single string "Mobile No. & Password combination do not match.". The previous gateway also used this exact wording on the unified failure path.
    Why Prevents account enumeration.
    Impact on you If your client was string-matching distinct messages (e.g., "User blocked"), the unified message replaces them.
    Action Match on the unified string, or — better — match on the body code field.
    Blocker? ❌ No. The previous gateway already used the same string on the joined credential check.

    Δ-4 — fetch-payment-status does the amount match at the database¶
    What changed The match is now WHERE merchant_id = ? AND order_id = ? AND amount = ? at the SQL level. The previous gateway behaved the same; an intermediate Java version compared amounts in memory after a broader fetch. The current engine restores the legacy behaviour.
    Why Deterministic, indexed lookup. Eliminates a class of decimal-precision edge cases.
    Impact on you If you were sending a wrong amount, you now consistently receive "Awaiting payment...". There is no scenario where the previous gateway succeeded and the current one fails for the same (order, amount).
    Action None for correct integrations.
    Blocker? ❌ No. Parity with the previous gateway.

    Δ-5 — validate-payment returns code 422 on "order not found"¶
    What changed Previously some Java builds returned code 404 for "order not found"; the current engine returns code 422 with the message "Sorry! Invalid Order ID.", matching the previous gateway's body-code shape.
    Why Restores legacy body-code parity.
    Impact on you A client that switched on code == 404 for "order not found" must now switch on code == 422 (with the messages[] text identifying the case).
    Action Switch on code for credential errors (422 + "Sorry! Invalid Store Credentials.") and for missing-order (422 + "Sorry! Invalid Order ID.").
    Blocker? ⚠️ Audit your client. Most legacy clients already expect 422 because that is what the previous gateway returned. If your client coded against an interim Java version, update the branch.

    Δ-6 — Per-field enumeration probes removed¶
    What changed The previous gateway exposed a sequence of Laravel-style validator probes — for example, a field-level message saying the merchant mobile does not exist before the credential check. The current engine returns a generic "Sorry! Invalid Store Credentials." instead.
    Why The earlier behaviour leaked whether a given mobile number or store password existed anywhere in the system, which is an enumeration vector.
    Impact on you A client that string-matched the field-specific message ("The merchant mobile no does not exist.") will not see it.
    Action Match on the unified credential message. If your business workflow requires the distinct case (very unlikely), contact integrations support — a feature-flagged legacy mode exists for the transition.
    Blocker? ⚠️ Audit your client. This is a deliberate security improvement. If you find a hard dependency on the field-specific message, raise it before go-live.

    Δ-7 — Activity log row written on successful direct payment¶
    What changed After a successful direct payment, the engine writes an audit row to activity_logs (mirroring the legacy ActivityDone event).
    Why Compliance and audit completeness — closes a gap that existed in early Java builds.
    Impact on you None at the API level. The row is internal. Your client will not observe this.
    Action None.
    Blocker? ❌ No. Internal only.

    8.3 New capabilities (opt-in, additive)¶

    These are not present on the previous gateway. They are available immediately on the current engine. None of them break existing integrations.

    Capability What it gives you How to adopt
    Canonical /api/v3/merchant/... paths Versioned, explicit prefix Use the new paths in new client code; old paths still work
    POST /merchant/generate-qr-token Same token + a server-generated QR URL + expires_in_minutes Replace generate-payment-token for in-store / kiosk / printed-media flows
    GET /merchant/fetch-payment-status & GET /merchant/fetch-payment-token-data Idempotent GET variants of the POST endpoints Use for retry-friendly polling and CORS-enabled browser inspection
    Signed IPN (X-ZiCharge-Signature + X-ZiCharge-Timestamp + X-ZiCharge-Request-Id) HMAC-SHA256 authenticity proof on every IPN; constant-time verification on your side Enable on the merchant dashboard; configure ipn_secret
    IPN retry with cooldown gate At-least-once delivery with bounded duplicate risk; up to 8 retries per terminal event Make your listener idempotent on X-ZiCharge-Request-Id
    Cash-back idempotency on reference_id Replay-safe cash-back / refunds Always supply a stable reference_id
    SSRF-hardened IPN dispatch Internal-range URLs are refused None — keep your ipn_url public-resolvable HTTPS
    Stronger token state machine Pessimistic lock + conditional UPDATE eliminates the historical "race between cancel and pay" class of bugs None — automatic
    First-page hosted UX polish Localized (en/ar/ku), responsive, iframe-embeddable, JS countdown, in-page cancel None — automatic
    Structured observability Every interaction tagged with X-ZiCharge-Request-Id; correlatable end-to-end Log the ID on your side, include in support requests

    8.4 Blocker check — explicit verdict¶

    The audit team compared the current Java implementation, endpoint by endpoint, against the documented legacy behaviour and the previous gateway's PHP source. Findings:

    Concern Verdict
    Endpoint paths drift ✅ None. All legacy paths preserved; canonical /api/v3/merchant/... is additive.
    Request shapes drift ✅ None. Same field names, same types, same required/optional split.
    Response envelope drift ✅ None. { messages, data, code } for v3 endpoints; legacy envelopes preserved for generate-payment-token and payment/direct.
    HTTP status drift ✅ None. HTTP 200 across the board, outcome in body code.
    IPN contract drift ✅ None. Form-encoded POST, same fields, same values. New optional headers (signature, timestamp, request-id) are additive.
    Token lifetime drift ✅ None. 15-minute validity preserved.
    Validation drift ⚠️ Two intentional changes — Δ-1 (regex on payment endpoints — matches previous gateway), Δ-2 (min bill_amount at mint — matches the documented limit). Neither is a regression.
    Body-code drift ⚠️ One intentional change — Δ-5 (validate-payment returns 422 for missing order, matching the previous gateway). Only an interim Java version returned 404.
    Message wording drift ⚠️ One intentional change — Δ-6 (per-field enumeration messages removed; unified credential message used). Deliberate security improvement.
    Side-effect drift ✅ One additive change — Δ-7 (audit-log row written on direct-payment success). Not externally observable.

    Verdict: no blockers for live merchants. All differences are either parity restorations against the previous gateway or deliberate, documented security improvements. The two items worth client-side auditing are Δ-5 (body code 422 on missing-order) and Δ-6 (unified credential message). Both can be handled by switching client logic from string-match to code-match — the recommended path even on the previous gateway.

    8.5 Migration plan template¶

    A safe path for an existing merchant to upgrade:

    1. Day 0 — point a sandbox client at https://dev.zicharge.com using the current engine. Run the test plan.
    2. Day 0–7 — audit your client for: hard-coded HTTP status checks (none should exist; if any do, switch to code), and for the two intentional message changes in Δ-5 and Δ-6.
    3. Day 7 — enable the signed-IPN feature on your sandbox merchant config; update your listener to verify signatures.
    4. Day 14 — sign off on the sandbox plan. Schedule production switch.
    5. Day 15 — flip production traffic. No DNS or endpoint change required — production already runs on the current engine. The "switch" is your own client deploy if any changes were needed.
    6. Day 15–30 — monitor IPN delivery success rate and reconciliation gaps using X-ZiCharge-Request-Id correlation.
